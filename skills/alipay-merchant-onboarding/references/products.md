## MCC 类目格式

MCC 类目通过读取 `references/mcc-reference.md` 进行语义和业务相关性匹配推荐。

### 返回格式

```json
{
  "recommendations": [
    {
      "level1_code": "A0001",
      "level1_name": "餐饮",
      "level2_code": "B0001",
      "level2_name": "中式正餐",
      "match_reason": "匹配原因"
    }
  ]
}
```

### mccCode 拼接规则

**格式：** `{level1_code}_{level2_code}`

**示例：** `A0001_B0001`

### 写入状态文件

```bash
MCC_CODE="${level1_code}_${level2_code}"
# ⚠️ mccCode 不写入共享内存状态文件，作为运行时变量使用
# 在签约提交时直接传入 apply JSON 的 businessProperty.mccCode
```

## 产品选择流程

### 2.0 智能推荐（优先执行）

**⚠️ 重要：方案规划阶段一次性推荐产品和经营类目，用户不满意时引导补充描述重新推荐。**

#### 智能推荐逻辑

```
1. 分析用户上下文（多来源综合判断）
   ├─ 用户对话描述（如"我是做AI工具的"）
   ├─ 项目名称/目录名（如 my-ai-agent、web-shop）
   ├─ 代码内容分析（依赖、API调用、业务逻辑）
   └─ 关键词匹配 → 推荐产品和MCC

2. 综合判断 → 输出推荐结果（产品 + 经营类目）
   └─ 展示完整推荐方案给用户确认（4个选项）
       ├─ 认可 → 保存方案，进入 Step 3
       ├─ 修改产品和类目 → 引导补充描述，重新推荐
       ├─ 仅修改产品 → 引导补充描述，仅重新推荐产品
       └─ 仅修改类目 → 引导补充描述，仅重新推荐类目
```

#### 上下文来源（优先级从高到低）

| 优先级 | 来源 | 说明 | 示例 |
|-------|------|------|------|
| 1 | 用户对话描述 | 用户主动说明业务场景 | "我是做AI智能体的" → AI收 |
| 2 | 项目名称/目录名 | 从工作目录推断业务类型 | `my-ai-assistant` → AI收 |
| 3 | 代码内容分析 | 分析项目依赖和代码特征 | 见下方详细规则 |

#### 代码内容分析规则

**通过 Glob/Grep 工具分析项目代码，判断业务场景：**

##### AI收特征（推荐AI收）

| 检测项 | 匹配规则 | 示例 |
|-------|---------|------|
| 依赖文件 | `package.json` 包含 AI/LLM 相关依赖 | `openai`, `anthropic`, `langchain`, `llamaindex` |
| AI Agent 框架 | 代码引用 Agent 框架 | `from langchain.agents`, `Agent`, `ChatOpenAI` |
| API 服务代码 | 代码包含 API 服务定义 | `@app.route`, `FastAPI`, `express.Router` |
| 按调用付费模式 | 代码包含计费/配额逻辑 | `credits`, `quota`, `usage`, `billing` |
| 算力相关 | 项目涉及模型推理 | `inference`, `model`, `gpu`, `cuda` |

**推荐场景示例：**

```
检测到项目特征：
  - package.json 包含 "openai" 依赖
  - 代码包含 Agent 相关类
  - 存在 API 路由定义

→ 推荐产品：AI收
→ 推荐理由：项目涉及 AI/智能体服务，适合AI收的机器支付场景
```

##### 电脑网站支付特征（推荐电脑网站支付）

| 检测项 | 匹配规则 | 示例 |
|-------|---------|------|
| 前端框架 | 传统 Web 框架 | `React`, `Vue`, `Next.js` (非 API 模式) |
| 电商相关依赖 | 购物车/支付相关 | `stripe`, `paypal`, `checkout`, `cart` |
| PC 网站特征 | 非 H5、非小程序 | 桌面端样式、响应式布局 |
| 政务/教育场景 | 业务逻辑特征 | `course`, `lesson`, `payment`, `order` |

**推荐场景示例：**

```
检测到项目特征：
  - package.json 包含 "react", "next.js" 依赖
  - 存在购物车/订单相关代码
  - 项目名称为 "my-ecommerce-site"

→ 推荐产品：电脑网站支付
→ 推荐理由：项目为 PC 端电商网站，适合电脑网站支付场景
```

#### 用户上下文分析

**关键词匹配规则（按优先级）：**

| 业务关键词 | 推荐产品 | 推荐MCC | 匹配理由 |
|-----------|---------|--------|---------|
| AI、智能体、Agent、API、算力、数字内容、按调用付费、机器支付 | AI收 | 根据业务匹配（如：互联网垂直电商平台 A0002_B0115） | AI收定义：面向AI智能体的机器支付 |
| 网站、电商、PC官网、在线教育、政务缴费、商城、PC网站 | 电脑网站支付 | 互联网综合电商平台 (A0002_B0114) 或匹配具体业务 | 电脑网站支付定义：PC网页发起支付 |
| 直播、视频、内容创作、数字内容 | AI收 | 商业生活服务 > 在线工具 (A0003_B0112) | 数字内容场景适合AI收 |
| SaaS、工具、软件服务、API服务 | AI收 | 商业生活服务 > 在线工具 (A0003_B0112) | API调用场景适合AI收 |

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

请确认方案是否合适？确认后将继续下一步。
  • 回复"确认"或"OK"继续
  • 如需调整产品类型或经营类目，请告知具体需求
```

**步骤 2：处理用户反馈**

| 用户反馈 | 处理方式 |
|---------|---------|
| "确认"、"OK"、"可以"、"没问题"、"好的" | 保存方案，进入 Step 3 登录授权 |
| "不对"、"不合适"、"换一个"、"不是这个" | 进入调整模式，提供同类目下最多 3 个备选 |
| 提供具体调整需求（如"我是做餐饮的"） | 根据需求重新匹配，再次输出推荐并等待确认 |

**调整模式输出格式（仅在用户不认可时触发）：**

```
📋 了解，以下是为您推荐的其他选项：

| 序号 | 经营类目 | 适用场景 |
|------|----------|----------|
| 1 | 餐饮 > 中式快餐 | 面条、盖饭等简餐 |
| 2 | 餐饮 > 饮品/甜品 | 奶茶、饮品店铺 |
| 3 | 餐饮 > 小吃/熟食 | 熟食、点心铺 |

请选择序号，或继续描述您的经营场景以便更精准匹配。
```

**用户选择后处理：**

| 用户输入 | 处理方式 |
|----------|---------|
| 输入数字序号（1/2/3） | 选择对应的类目，保存方案，进入 Step 3 登录授权 |
| 输入更详细描述 | 根据新描述重新匹配，返回新的推荐结果 |
| 输入"确认"（单推荐时） | 确认使用推荐的类目，保存方案，进入 Step 3 登录授权 |

## 资料采集需求

### 电脑网站支付

| 序号 | 资料名称 | 说明 |
|------|----------|------|
| 1 | 首页截图 | 网站首页截图 |
| 2 | 商品页截图 | 商品或服务页面截图 |
| 3 | 支付页截图 | 支付页面截图 |

**格式要求：** JPG/PNG，≤10MB/张

### AI收

**仅在服务注册阶段交互式收集（Step 5.2 服务注册），无需预先采集。**

服务注册所需信息如下（详见 `references/service-registration.md`）：

| 序号 | 资料名称 | 说明 |
|------|----------|------|
| 1 | 服务名称 | 服务名称，如"天气查询"，长度 1-50 字符 |
| 2 | 服务描述 | 简要描述服务功能，长度 1-500 字符 |
| 3 | 服务地址 | 可访问的 API 地址 URL |
| 4 | 服务单价 | 服务单价（元），最低 0.01 元 |
| 5 | 请求示例 | JSON 格式的接口请求示例 |

**采集时机：** AI收无独立资料采集步骤（Step 4 直接标记 completed）。以上 5 项信息在 Step 5.2 服务市场注册阶段交互式收集，不存入共享内存状态文件。
