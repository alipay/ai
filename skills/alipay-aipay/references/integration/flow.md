## ⛔ 强制执行要点

### 强阻塞阅读项，必须遵循，否则违规！

1. **先定计划再动手**：执行任何操作之前，**必须先确定功能分支并打印完整的待办步骤清单**，禁止未确定计划就直接开始执行
2. **必须获得用户同意**：未经用户同意，禁止启动集成流程
3. **单一产品原则**：一次只能集成一个产品，完成后再讨论下一个
4. **禁止假数据**：禁止编造沙箱配置或使用占位符
5. **阻塞确认点**：必须等待用户明确回复后才能继续下一步
6. **禁止合理化跳过确认**：禁止用"用户可能希望快速完成"、"这是一个简单场景"等理由跳过阻塞确认点
7. **禁止假数据与占位符冒充**：沙箱成功返回完整配置时，禁止用占位符代替真实返回值；禁止自行生成示例私钥、示例账号等假数据
8. **先阅读文档再执行**：每个步骤必须先阅读相关文档，完成该步骤的**完成条件**自检后，方可进入下一步
9. **生产密钥替换提醒**：当用户要求直接替换生产环境密钥配置，或声称已替换完成并要求启动生产环境支付测试时，必须先基于项目编程语言和应用私钥格式提醒用户：`appId`、应用公钥、应用私钥必须属于同一套应用密钥；应用私钥格式必须与语言 SDK 匹配，格式不匹配时使用支付宝开放平台密钥工具做私钥格式转换。

### 步骤 2：沙箱初始化

**任务**：创建沙箱测试环境

> ⚠️ **前置要求**：在调用沙箱工具前，必须先按 `../normal/alipay-cli-env.md` 完成 alipay-cli 检测与安装
>
> ⚠️ **必须完整阅读** `modules/sandbox/sandbox-setup-guide.md` 后方可执行本步骤

**完成条件**：
- [M-02] **必须**已完整阅读 `modules/sandbox/sandbox-setup-guide.md`（含强校验规则、输出格式、红线禁止）
- [M-02] 已检测用户操作系统类型
- [M-02] Unix/macOS/Linux：已调用沙箱工具成功创建环境
- [M-02] Windows：已引导用户自行获取沙箱信息
- [M-02] 已按文档要求完成沙箱信息强校验（**逐项核对必含字段清单**），并完整输出

**操作流程**：
1. 检测用户操作系统（Windows/macOS/Linux）
2. Windows用户：引导自行获取沙箱信息，参考 `modules/sandbox/sandbox-setup-guide.md` 第 1.2 节
3. Unix/macOS/Linux用户：调用沙箱工具创建环境

**⚠️ 沙箱信息输出格式（必须严格遵循）**：

> 详细输出格式见 `modules/sandbox/sandbox-setup-guide.md` 第 3.2 节
>
> **必须按表格格式输出**，包含以下全部内容：

| 类别 | 必含字段 |
|------|---------|
| 应用配置 | `appId`、`appPrivateKey`、`appPrivatePkcsKey`、`alipayPublicKey`、`appPublicKey`、`pid` |
| 商家账号 | `accountDesc`、`email`、`merchantId`、`userId`、`acctrans` |
| 买家账号 | `accountDesc`、`email`、`userName`、`userId`、`logonPassword`、`payPassword` |
| 沙箱信息 | `sandboxId`、`sandboxName` |

**强校验规则**：
- 工具返回后，**必须逐项核对**上述必含字段是否存在且非空
- 如有字段缺失，**阻断流程**，要求重新创建
- 后续步骤将沙箱配置写入用户项目后，必须做配置准确性后置校验，确认写入值来自同一份沙箱返回配置，且与沙箱返回值一致、字段选择正确、SDK 配置匹配、无占位符或截断。校验失败必须阻断后续联调，先重新复制沙箱字段或修正 SDK 配置。

**绝对禁止**：
- ❌ 禁止编造字段
- ❌ 禁止用占位符（xxx）代替真实返回值
- ❌ 禁止用示例假数据（如 `MIIEvw...`）冒充
- ❌ 禁止使用 JSON 代码块输出
- ❌ 禁止省略任何字段

> ⚠️ 详细操作、输出格式、强校验规则、红线禁止请参阅 `modules/sandbox/sandbox-setup-guide.md`

**🚫 SDK 防幻觉强制规则**：

> ⚠️ 以下规则来自真实集成踩坑，违反将导致集成失败。详细说明和各语言操作对照见 [SDK 说明文档](modules/alipay-sdk-reminder.md)，**严禁**在未读取该文档的情况下生成代码或给出建议。

| # | 规则 | 禁令 |
|---|------|------|
| 1 | 私钥字段按语言选择沙箱返回值 | Java 使用 `appPrivateKey`（PKCS#8），非 Java 使用 `appPrivatePkcsKey`（PKCS#1），禁止自行生成或格式转换 |
| 2 | 页面跳转类 API 必须使用页面跳转方法 | `alipay.trade.page.pay` 等必须使用 `pageExecute()`/`pageExec()`，使用 `exec()` 将无法获取支付表单 |
| 3 | 前端禁止用 URL 直接跳转支付表单 | 支付接口返回的是 HTML 表单，必须渲染并自动提交 form，直接用 URL 跳转会导致页面只显示参数而无法跳转支付宝 |
| 4 | 时间戳格式 | 必须使用 `yyyy-MM-dd HH:mm:ss`，禁止使用 ISO 格式 |
| 5 | SDK 引入方式 | 必须通过查阅 SDK 文档或类型定义确定正确的引入方式，禁止凭猜测选择 |
| 6 | 遇到 `invalid-signature` 报错 | 严禁凭猜测归因到私钥格式，最常见原因是请求参数不完整（如 `return_url` 未传入） |
| 7 | 配置写入后必须后置校验 | 必须确认写入配置来自同一份沙箱返回配置、与沙箱返回配置一致、字段映射正确、SDK 配置匹配；失败时禁止继续联调 |

6. 沙箱配置：无编造/假数据？
7. 代码示例：已查阅当前语言的产品类别 + 通用接口？
8. 配置后置校验：是否已确认写入配置来自同一份沙箱返回配置、与沙箱返回配置一致、字段映射正确、SDK 配置匹配？

**按量付费生产实现边界**：
- 返回 `Payment-Needed` 前必须把 `out_trade_no`、`resource_id`、金额、订单状态和过期时间持久化到商户数据库。内存仓储只适合本地临时演示，服务重启后旧付款链接无法继续履约。
- `alipay.aipay.agent.payment.verify` 成功只代表支付宝支付凭证有效；交付资源前仍必须映射回商户本地订单，并校验订单状态、金额、资源归属和幂等履约状态。
- 沙箱联调中 `outTradeNo`、`resourceId` 等验付响应字段可能出现空字符串，排查时先判断字段是否非空再定位问题；生产环境不得因字段为空而跳过核心校验，应按异常凭证或异常订单处理。
- 资源生成后先保存履约结果为可重试状态，再调用 `alipay.aipay.agent.fulfillment.confirm`。确认成功后才标记订单为 `FULFILLED` 并返回成功交付；确认失败时保留结果并允许同一笔 `Payment-Proof` 重试。

> ⚠️ **按量付费**：代码生成后请启动本地服务，后续步骤 5 将进行端到端沙箱支付测试。服务需实现 402 协议（无 Payment-Proof 时返回 HTTP 402 + Payment-Needed 头）。

### 步骤 5：按量付费沙箱测试（仅按量付费）

## 问题排查流程（功能二）

**触发条件**：用户在集成支付宝支付产品过程中遇到报错或其他问题。触发关键词："报错"、"错误码"、"问题"、"排查"、"调试"、"异常"等。

执行下述问题排查步骤之前，**必须明确用户当前集成的支付产品**（按量付费/网站支付/APP支付），否则暂停输出并要求用户澄清。

### 问题识别与分流

根据用户输入判断问题类型，分流到对应排查路径：

```
用户问题
    |
    +-- 验签失败（invalid-signature / 验签出错）  ← 优先匹配
    |       |
    |       └───> 常见问题排查（验签失败专项）
    |
    +-- 有明确错误码（如 "ACQ.TRADE_HAS_SUCCESS"、"INVALID_PARAMETER"）
    |       |
    |       └───> 错误码排查
    |
    +-- 无明确错误码（如流程疑问、功能异常等）
            |
            └───> 常见问题排查
```

### 错误码排查

**适用**：用户提供了明确的错误码。

错误码查询前，**必须确认发生报错的接口信息**，否则暂停输出并要求用户澄清。

#### 排查流程

1. **查公共错误码**：查阅 [公共错误码说明](https://ideservice.alipay.com/cms/site/02km9f)，根据用户提供的错误码检索相关内容。如有匹配结果，**输出排查结论**；否则，**查业务错误码**。

2. **查业务错误码**：基于确定的支付产品和报错接口信息，查阅对应产品文档，在接口文档中根据错误码检索相关内容。

3. **输出排查结论**：根据查询到的错误码关联内容，输出排查结论。

### 常见问题排查

**适用**：无明确错误码的其他类型问题，或错误码为 `invalid-signature`（验签出错）。

#### 排查流程

1. 根据用户输入和确定的支付产品，查阅对应的产品**常见问题文档**，匹配问题解决方案。

2. 若未找到解决方案，引导用户查阅 [开放平台在线文档](https://open.alipay.com?form=payskill) 或咨询 [支付宝技术支持](https://opensupport.alipay.com/support/intelligent-services?form=payskill)，**严禁**编造解决方案。
3. 对代码集成、SDK 使用、应用创建、应用发布等没有确定依据的问题，必须明确说明无法确认，并引导用户前往支付宝开放平台或支付宝技术支持咨询客服。

#### 常见问题文档索引

| 产品类别 | 常见问题文档 |
|----------|-------------|
| 按量付费 | [按量付费常见问题](https://ideservice.alipay.com/cms/site/0j7uos) |
| 网站支付 | [网站支付常见问题](https://ideservice.alipay.com/cms/site/0j3kh1) |
| APP支付 | [APP支付常见问题](https://ideservice.alipay.com/cms/site/0j3pih) |

#### ⛔ 验签失败（invalid-signature）专项排查

遇到 `invalid-signature`（验签出错）时，**严禁凭猜测归因到私钥格式或环境兼容性**，必须按以下原则排查：

1. **先验证实际请求内容，再怀疑密钥配置** — 验签失败最常见的原因是请求参数不完整或不匹配（如 `return_url` 未传入），而非私钥格式问题。

2. **严禁陷入连环误判**：看到 `ERR_OSSL_UNSUPPORTED` 就推断私钥格式有问题 → 给私钥添加 PEM 头尾或其他前后缀 → 转换格式，这是错误排查路径。详见 `modules/alipay-sdk-reminder.md` 中的「验签失败排查原则」。

3. **正确排查顺序**：
   - 检查请求参数完整性（最常见原因）
   - 检查私钥字段是否正确（Java 用 `appPrivateKey`，其他语言用 `appPrivatePkcsKey`）
   - 检查 signType 是否为 RSA2
   - 检查网关地址是否正确

4. **线上环境公私钥不匹配**：若线上环境报错"公私钥不匹配"，建议用户：
   - 登录支付宝开放平台**重新上传应用公钥**
   - 在应用配置中重新配置应用公私钥对
