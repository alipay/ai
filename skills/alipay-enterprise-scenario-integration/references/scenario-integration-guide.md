# 企业码单场景接入指南

本文件承接 `SKILL.md` 的总控流程，放置集成准备、默认方案、场景交付和验证要求。它不是完整的代码生成流程；代码生成前必须回到 `SKILL.md`，先读取 [多 Agent 代码生成编排规则](multi-agent-codegen.md) 和 [主方案聚合质量门禁](quality-gates/aggregate.md)，再执行 SDK 预检、分域生成和主聚合校验。场景枚举与规则因子以子 Skill 文档为事实源，具体决策见 [场景决策规则](scenario-decision-rules.md)。

## 集成准备

### 账号及产品权限

需要准备两个企业支付宝账号：

| 账号类型 | 说明 |
|---|---|
| 服务商账号 | 作为服务商角色 |
| 企业账号 | 由服务商引导完成企业码企业入驻，并预充资金供员工因公支付时代付扣款 |

[注册企业支付宝账号并完成企业认证](https://opendocs.alipay.com/common/02kkum)

### 开发环境

1. [完成开放平台应用配置](https://ideservice.alipay.com/cms/site/0izal1.md)
2. [下载服务端 SDK](https://ideservice.alipay.com/cms/site/0j0cjj)
3. [接口调用配置](https://ideservice.alipay.com/cms/site/0izal1.md)

接口通用参数 `charset` 使用 `utf-8`。

读取 `https://ideservice.alipay.com/cms/site/{文档ID}` 时使用 `curl -sL` 跟随重定向。代码生成仍需按 [主方案聚合质量门禁](quality-gates/aggregate.md) 独立确认实际 SDK 来源、版本和可用类。

## 子 Skill

本方案引用同级三个子 Skill：

| 能力域 | 子 Skill | 默认范围 |
|---|---|---|
| 员企 | [员企 Skill](../../alipay-enterprise-ec/SKILL.md) | 企业入驻、员工签约、企业管理、员工管理 |
| 费控 | [费控 Skill](../../alipay-enterprise-expense-control/SKILL.md) | 制度管理 |
| 账单 | [账单 Skill](../../alipay-enterprise-bill/SKILL.md) | 推模式账单管理 |

部门、核算主体、企业地址、企业消息任务、手工发放、额度管理、订单同步和对账单下载按用户需求纳入；如果所选场景依赖订单履约明细，订单同步应纳入范围，是否必选以账单子 Skill 文档为准。用户只选基础模块时，不读取或实现扩展模块。

方案 Skill 启动后、读取本指南的场景决策内容之前执行：

```bash
node alipay-enterprise-scenario-integration/tools/install_subskills.js
```

安装器会把缺失的子 Skill 解压到方案 Skill 的平级目录并校验完整性。方案设计和代码生成都需要读取子 Skill 的场景、枚举与规则文档，因此安装失败时必须在场景决策前阻断，不把解压步骤留给接入方手工处理。安装完成只代表文件存在；代码生成仍必须按 [多 Agent 代码生成编排规则](multi-agent-codegen.md) 显式加载子 Skill。

## 场景决策

每次任务只接入一个费用场景。代码生成前必须：

1. 从费控子 Skill 的费用类型、子类和约束文档确认合法组合。
2. 按默认策略确定因公场景：未明确时使用“默认”（接口值 `DEFAULT`），票务类默认使用“差旅”（接口值 `TRAVEL`），用户或上下文明确其它因公场景时才使用对应枚举。
3. 确认全部必用规则因子及其业务值。
4. 如选择内部费控，确定制度额度/发放来源：默认使用发放规则；用户或上下文明确需要制度可用额度限额条件、手工发放接口时，才切换到对应模式并确认所需业务值。
5. 因公优先默认关闭；只有用户明确提出需要时，才判断是否适用并继续确认规则值。用户未提及时不主动询问、不作为确认选项、不生成因公优先规则组合。
6. 确认账单侧用于识别该场景的字段。
7. 写入 `.alipay-skill/scenario.json`。

上下文可唯一推断时展示决策后继续；存在多个合法值或缺少业务值时必须询问。问询只覆盖未决项，不得把已确认的费控模式、模块范围或默认策略重新混入选项。不得为了继续生成而把费用类型、费用子类、规则因子或业务值填成 `DEFAULT`、空字符串或猜测值；因公场景只按本节默认策略取值。

具体费用场景的必用因子和值以费控子 Skill 的约束文档和本域 validator 为准；本方案层只负责沿用已确认决策并完成跨域聚合。因公优先等跨域决策见 [场景决策规则](scenario-decision-rules.md)。

## 代码生成边界

- 新工程可以创建公共骨架，但业务代码仍按员企、费控、账单分域生成。
- 已有项目第一轮只做盘点、增量计划和拟定衔接契约；用户确认后才修改代码。
- 已有项目优先复用现有 SDK 配置、通知入口和业务对象，不得生成与现有系统脱节的旁路模块。
- 所有字段、枚举和接口参数必须来自子 Skill 引用文档；文档未定义时明确说明，不自行补齐。
- 公共配置、共享消息入口和跨域冲突由主 Agent 统一处理。

代码生成阶段的证据表、子 Agent 分工、单 Agent 降级和完成门禁见 [多 Agent 代码生成编排规则](multi-agent-codegen.md) 与 [主方案聚合质量门禁](quality-gates/aggregate.md)。本指南只要求方案输出说明已确认的场景、模块和仍阻塞实现的缺口。

## 输出要求

### 方案设计

必须包含：

1. 场景决策：费用类型、子类、因公场景、必用规则因子及已确认值。
2. 模式决策：员企、费控、账单和消息接入方式。
3. 内部费控制度额度/发放来源决策：默认发放规则，或用户明确选择的制度可用额度限额条件/手工发放接口。
4. 模块与接口清单：必选、按需和未纳入项。
5. 因公优先仅在用户主动提出时输出规则配置；默认关闭时不作为待确认项。
6. 账单场景识别方式。
7. 仅列出仍会阻塞实现的待确认项。
8. 端到端验证要点。

### 代码生成

除方案设计内容外，还必须包含：

1. 多 Agent 启动状态和子 Skill 加载握手。
2. `.alipay-skill/scenario.json`。
3. 各域接口证据表、生成文件、覆盖范围和本域自检结果。
4. 场景制度配置、账单识别、通知处理和主聚合校验结果。

已有项目还需列出 baseline、复用能力、实际新增/修改文件、未触碰公共文件和衔接契约执行结果。

## 交付验证

1. 所选三域模块的接口、通知和必要业务链路已覆盖。
2. 费控制度、账单识别和通知处理与 `.alipay-skill/scenario.json` 一致。
3. 业务值真实落入生成代码，不能只存在于常量、文档或枚举中。
4. 生成环境通过子域 validator、主聚合 validator 以及可执行的构建/测试；这些脚本不是接入方工程运行依赖。
