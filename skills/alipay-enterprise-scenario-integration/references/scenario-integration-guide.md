# 企业码单场景接入指南

本文件承接 `SKILL.md` 的总控流程，放置集成准备、默认方案、场景交付和验证要求。场景枚举与规则因子以子 Skill 文档为事实源，具体决策见 [场景决策规则](scenario-decision-rules.md)。

## 集成准备

### 账号及产品权限

需要准备两个企业支付宝账号：

| 账号类型 | 说明 |
|---|---|
| 服务商账号 | 作为服务商角色 |
| 企业账号 | 由服务商引导完成企业码企业入驻，并预充资金供员工因公支付时代付扣款 |

[注册企业支付宝账号并完成企业认证](https://opendocs.alipay.com/common/02kkum)

### 开发环境

1. [完成开放平台应用配置](https://ideservice.alipay.com/cms/site/0izal1.md)
2. [下载服务端 SDK](https://ideservice.alipay.com/cms/site/0j0cjj)
3. [接口调用配置](https://ideservice.alipay.com/cms/site/0izal1.md)

接口通用参数 `charset` 使用 `utf-8`。

读取 `https://ideservice.alipay.com/cms/site/{文档ID}` 时使用 `curl -sL` 跟随重定向。代码生成仍需按 [主方案聚合质量门禁](quality-gates/aggregate.md) 独立确认实际 SDK 来源、版本和可用类。

## 子 Skill

本方案引用同级三个子 Skill：

| 能力域 | 子 Skill | 默认范围 |
|---|---|---|
| 员企 | [员企 Skill](../../alipay-enterprise-ec/SKILL.md) | 企业入驻、员工签约、企业管理、员工管理 |
| 费控 | [费控 Skill](../../alipay-enterprise-expense-control/SKILL.md) | 制度管理 |
| 账单 | [账单 Skill](../../alipay-enterprise-bill/SKILL.md) | 推模式账单管理 |

部门、核算主体、企业地址、企业消息任务、手工发放、额度管理、订单同步和对账单下载按用户需求纳入。用户只选基础模块时，不读取或实现扩展模块。

同级子 Skill 缺失时，可从本 Skill 的 `subskills/*.zip` 解压。安装只代表文件存在；代码生成仍必须按 [多 Agent 代码生成编排规则](multi-agent-codegen.md) 显式加载子 Skill。

## 场景决策

每次任务只接入一个费用场景。代码生成前必须：

1. 从费控子 Skill 的费用类型、子类和约束文档确认合法组合。
2. 确认 `scene_type`。
3. 确认全部必用规则因子及其业务值。
4. 判断因公优先是否适用；淘系平台场景不支持，直接关闭且不询问，其他场景再按需确认。
5. 确认账单侧用于识别该场景的字段。
6. 写入 `.alipay-skill/scenario.json`。

上下文可唯一推断时展示决策后继续；存在多个合法值或缺少业务值时必须询问。不得为了继续生成而填 `DEFAULT`、空字符串或猜测值。

`TICKET/TICKET`、因公优先等跨方案业务规则见 [场景决策规则](scenario-decision-rules.md)，其底层事实同时下沉到费控子 Skill，确保子 Skill 独立使用时仍成立。

## 代码生成边界

- 新工程可以创建公共骨架，但业务代码仍按员企、费控、账单分域生成。
- 已有项目第一轮只做盘点、增量计划和拟定衔接契约；用户确认后才修改代码。
- 已有项目优先复用现有 SDK 配置、通知入口和业务对象，不得生成与现有系统脱节的旁路模块。
- 所有字段、枚举和接口参数必须来自子 Skill 引用文档；文档未定义时明确说明，不自行补齐。
- 公共配置、共享消息入口和跨域冲突由主 Agent 统一处理。

## 输出要求

### 方案设计

必须包含：

1. 场景决策：费用类型、子类、`scene_type`、必用规则因子及已确认值。
2. 模式决策：员企、费控、账单和消息接入方式。
3. 模块与接口清单：必选、按需和未纳入项。
4. 因公优先决策及其规则配置（如启用）。
5. 账单场景识别方式。
6. 仅列出仍会阻塞实现的待确认项。
7. 端到端验证要点。

### 代码生成

除方案设计内容外，还必须包含：

1. 多 Agent 启动状态和子 Skill 加载握手。
2. `.alipay-skill/scenario.json`。
3. 各域已读文档、生成文件、接口和通知覆盖、本域自检结果。
4. 场景制度配置代码和账单识别代码。
5. 通知验签、幂等、重复通知和失败重试处理。
6. 对应技术栈的构建、测试和主聚合校验结果。

已有项目还需列出 baseline、复用能力、实际新增/修改文件、未触碰公共文件和衔接契约执行结果。

## 交付验证

1. 企业入驻和员工签约链路可走通，并能处理对应通知。
2. 所选模块的完整接口与必要通知已覆盖。
3. 制度中的费用类型、子类、`scene_type`、规则因子和值与 `scenario.json` 一致。
4. 场景要求的业务值已真实落入制度规则，不能只存在于常量、文档或枚举中。
5. 账单通知和查询可识别所选场景，并解析必要扩展内容。
6. 消息链路具备验签、幂等、重复通知和失败重试处理。
7. 外部费控场景包含所选 SPI 能力的真实实现和测试。
8. 生成环境通过子域 validator 和主聚合 validator；这些脚本不是接入方工程运行依赖。
