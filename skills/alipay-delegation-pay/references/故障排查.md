# 故障排查

## 1. 签名相关

### 1.1 参与签名的参数口径

**核心原则：签名内容必须和你实际发送的请求参数完全一致——发什么字段就签什么字段，不能多签、也不能漏签。** 拼接方式：把要签名的参数按 key 字典序排序，`key=原始值` 用 `&` 连接，RSA-SHA256（RSA2）签名后 base64 编码。

`app_id`/`biz_content`/`charset`/`method`/`sign_type`/`timestamp` 这 6 个是任何请求都必带、因此必然参与签名的字段。但 `format`、`version` 等字段是否参与签名，取决于**你的请求里到底有没有发送它们**：

- 反编译官方 `alipay-sdk-java`（`AbstractAlipayClient` / `AlipaySignature`）确认：SDK 默认会把 `format`、`version`、`alipay_sdk` 等字段一并放进请求参数，并且**全部纳入签名计算**——用 `sdkExecute`/`execute` 的正常路径，这些字段该签就签，不用你操心。
- 如果你**不用官方 SDK**、手工拼参数发起请求：只要签名内容和实际发送的参数集合对不上（比如签了 `format` 但没在请求里发，或者反过来发了却没签），网关重新计算出的签名就会不一致，返回**无效签名**。

> **常见坑**：手工拼参数时，"图省事只发 6 个必填字段（不发 `format`/`version`）"是完全允许的——但**签名内容也必须同步只包含这 6 个字段**，多签一个没发送的字段一样会导致验签失败。下面的脚本示例就是这个用法：只发送、只签名这 6 个字段，两边保持一致。

### 1.2 零依赖本地生成签名串（用于调试/非 Java 技术栈）

如果暂时没有官方 SDK 环境，或想脱离业务代码单独验证签名逻辑，可以用下面的 Node.js 脚本本地生成 `alipay.user.agreement.delegation.apply` 的完整请求 URL（纯本地计算，不发起网络请求）：

```js
// gen_delegation_orderstr.js —— 把下面几个占位符换成你自己的配置后即可运行
const crypto = require('crypto');

const GATEWAY = 'https://openapi.alipay.com/gateway.do'; // 生产网关；沙箱见 00-接入准备.md
const APP_ID = 'your_app_id';                             // 替换成你的 APPID
const APP_PRIVATE_KEY_PKCS8 = 'your_private_key_base64';  // 替换成你的应用私钥（PKCS8 base64，单行）

function toPem(b64) { return `-----BEGIN PRIVATE KEY-----\n${b64.match(/.{1,64}/g).join('\n')}\n-----END PRIVATE KEY-----\n`; }
function fmtTs(d) { const p = n => String(n).padStart(2, '0'); return `${d.getFullYear()}-${p(d.getMonth() + 1)}-${p(d.getDate())} ${p(d.getHours())}:${p(d.getMinutes())}:${p(d.getSeconds())}`; }
// 支付宝网关按 Java URLEncoder 风格编码：空格 -> +
function alipayEncode(v) { return encodeURIComponent(v).replace(/%20/g, '+').replace(/[!'()*]/g, c => '%' + c.charCodeAt(0).toString(16).toUpperCase()); }

const uniq = Date.now();
const bizContent = {
  agent_id: 'your_agent_id',
  personal_product_code: 'UAM_AGENT_AUTH_P',
  external_agreement_no: 'agreementno_' + uniq,   // 每次唯一
  conversation_history: [
    { role: 'USER', content: '帮我抢一张1月20号回哈尔滨的票，二等座。', create_time: '2025-12-23 10:28:00' },
  ],
  delegation_params: {
    delegation_tag: '定时购买',
    max_total_amount: '10',
    times_limit: '100',
    validity_start_time: '2026-06-26',              // 换成实际生效日期
    validity_end_time: '2026-07-26',
    external_delegation_id: 'delegation_' + uniq,    // 每次唯一
    delegation_desc: '监控1月20日北京去哈尔滨的高铁二等座',
  },
  access_params: { channel: 'ALIPAYAPP' },
};

// 公共参数：本脚本只发送这 6 个字段，所以也只对这 6 个字段签名（发什么签什么）。
// 如果你想额外带上 format/version 等字段，记得把它们也加进签名内容，否则会验签失败。
const params = {
  app_id: APP_ID,
  method: 'alipay.user.agreement.delegation.apply',
  charset: 'UTF-8',
  sign_type: 'RSA2',
  timestamp: fmtTs(new Date()),
  biz_content: JSON.stringify(bizContent),
};

const signContent = Object.keys(params).sort().map(k => `${k}=${params[k]}`).join('&');
params.sign = crypto.createSign('RSA-SHA256').update(signContent, 'utf8').sign(toPem(APP_PRIVATE_KEY_PKCS8), 'base64');
const query = Object.keys(params).sort().map(k => `${k}=${alipayEncode(params[k])}`).join('&');

console.log(`${GATEWAY}?${query}`); // 完整请求 URL
console.log(query);                 // 仅 orderStr（客户端 SDK 唤起用）
```

运行：

```bash
node gen_delegation_orderstr.js
```

注意事项：

- `timestamp` 有时效，生成后别放太久，过期需重新运行脚本。
- `external_agreement_no` / `external_delegation_id` 每次都要换新值（用时间戳保证唯一），避免复用旧协议导致签约异常。
- orderStr 生成后需要传给客户端 SDK（Android `payV2` / iOS `payOrder`）唤起支付宝完成授权。

### 1.3 验证签名口径的方法（排查"无效签名"的通用手段）

如果不确定自己的签名口径是否正确，可以拿一条**已知有效**的请求 URL（比如从日志、官方 Demo 或联调工具里拿到的），在本地反推验证：

1. 取出 URL `?` 后面的 query 部分。
2. form-decode（`+` 还原成空格）解析出各参数。
3. 去掉 `sign` 字段，剩余参数按 key 排序后拼接，用同样的私钥重新计算 RSA-SHA256 签名。
4. 与原始 `sign` 对比：一致则说明参数集合/拼接方式正确；不一致则说明签名口径（多带了字段、少带了字段、编码方式不对等）有问题。

```js
function formDecode(s) { return decodeURIComponent(s.replace(/\+/g, ' ')); }
```

## 2. 委托代买申请：会话历史不生效

`conversation_history` 内的时间字段官方文档标注为 `create_time`（下划线），但不同 SDK 版本对该字段的序列化可能存在差异。如果调用成功但用户在支付宝客户端看不到会话历史展示，可以尝试把该字段换成驼峰 `createTime` 再试一次，两种命名都测一遍以确认当前 SDK 版本的实际口径。

## 3. 其他排查建议

- 先确认环境是否对齐：生产 `appId` + 生产私钥 + 生产网关必须是同一套，沙箱同理，混用会导致验签失败。
- 检查是否已联系支付宝 BD 开通「智能体代理支付」邀测产品，未开通会返回 `ACCESS_FORBIDDEN`，容易和"无效签名"混淆，先看错误码/子错误码区分。
- 更多接入前置准备见 [00-接入准备.md](../00-接入准备.md)。
